## 1. 整体架构概览

Scene 的架构可以概括为 **"三层 + 四引擎"**：

```
┌────────────────────────────────────────────────────────┐
│                     UI 层（46 个 xml 布局 + 自定义 View）    │
│   Activity 界面  ·  悬浮窗  ·  桌面小部件  ·  设置磁贴      │
├────────────────────────────────────────────────────────┤
│                    逻辑层（vtools 模块）                  │
│   系统调校  ·  进程管理  ·  应用冻结  ·  性能监控  ·  脚本    │
├──────────────┬──────────────┬──────────────┬────────────┤
│  kr-script   │  场景模式     │  Xposed      │  原生代码   │
│  脚本引擎     │  自动切换     │  Hook 系统    │  JNI 调用   │
├──────────────┴──────────────┴──────────────┴────────────┤
│                    系统接口层                            │
│  ROOT Shell  ·  /proc 文件系统  ·  sysfs  ·  JNI  ·  Xposed │
└────────────────────────────────────────────────────────┘
```

### 架构特点

1. **混合权限模型**：同时依赖 ROOT、Xposed、AccessibilityService 三种方式获取系统级权限，互为补充
2. **脚本驱动调校**：核心调校逻辑写在 Shell 脚本中（kr-script），不由 Java 代码直接操作 sysfs
3. **离线优先**：大部分功能可离线使用，仅脚本市场/在线插件等模块需网络
4. **多厂商适配**：内置 MIUI / Flyme / OPPO / vivo / 通用 AOSP 等多种定制系统的对应脚本

## 2. kr-script 脚本引擎

这是 Scene 最核心的机制——通过 Shell 脚本执行系统调校。

### 2.1 引擎架构

```
kr-script.conf  ──→  入口：executor.sh
                           │
               ┌───────────┼───────────┐
               ▼           ▼           ▼
         more.xml    各厂商配置文件   各 .sh 脚本
        (页面导航)   (aosp/miui/    (141 个脚本)
                      oppo/vivo/...)
```

### 2.2 配置文件体系

| 文件 | 用途 |
|------|------|
| `kr-script.conf` | 引擎入口配置，指定 executor 和页面列表 |
| `more.xml` | 主页面菜单导航，组织功能入口 |
| `aosp/aosp.xml` | AOSP 通用功能页面 |
| `miui/miui.xml` | MIUI 定制功能（如云配置、温控）|
| `flyme/flyme.xml` | Flyme 定制功能 |
| `oppo/oppo.xml` | OPPO/一加 定制功能（含 ColorOS）|
| `vivo/vivo.xml` | vivo 定制功能（含 OriginOS）|
| `qualcomm/qualcomm.xml` | 高通平台特定功能 |
| `display/display.xml` | 显示相关设置（分辨率、刷新率等）|
| `battery/battery.xml` | 电池相关设置 |
| `developer/developer.xml` | 开发者选项相关 |
| `apps/apps.xml` | 应用相关设置 |
| `img/img.xml` | 镜像相关工具 |
| `other/other.xml` | 其他杂项功能 |

### 2.3 XML → UI 节点映射

kr-script 的 XML 配置描述了一套可交互的 UI 层级结构，Java 代码中的 `krscript.model` 包将其映射为节点对象：

| XML 元素 | Java 模型 | UI 表现 |
|----------|-----------|---------|
| `<page>` | `PageNode` | 新页面 |
| `<group>` | `GroupNode` | 分组标题 |
| `<action>` | `ActionNode` | 可点击项 → 执行脚本 |
| `<switch>` | `SwitchNode` | 开关控件 |
| `<picker>` | `PickerNode` | 选择器/下拉列表 |
| `<text>` | `TextNode` | 静态文本展示 |
| `<runnable>` | `RunnableNode` | 后台运行脚本 |
| `<clickable>` | `ClickableNode` | 点击触发项 |

### 2.4 厂商适配脚本

kr-script 内针对不同厂商做了专门的适配（脚本按厂商分类）：

| 厂商 | 覆盖范围（示例） |
|------|-----------------|
| **MIUI** | MIUI 云配置、温控策略、动画速度、广告屏蔽 |
| **Flyme** | 魅族系统设置 |
| **OPPO/一加** | 极致性能调度（`extreme_gt`）、ORMS |
| **vivo** | 蓝厂系统设置 |
| **Qualcomm** | 高通平台特性（如 GPU 调频）|
| **MTK** | 联发科平台特性 |
| **AOSP** | 通用 Android 设置 |

### 2.5 工具链

`assets/toolkit/` 目录提供脚本运行时所需的二进制工具：

| 工具 | 用途 |
|------|------|
| `busybox` | 嵌入式 Linux 工具箱（增强 Shell 功能）|
| `binder12.so` ~ `binder15.so` | Binder 代理工具（不同 SDK 版本）|
| `run` / `pipe` | 进程执行/管道工具 |
| 各类辅助 `.sh` | 环境检测、工具安装 |

## 3. 场景模式系统

### 3.1 触发途径

场景模式的切换有四种触发方式：

```
┌─ NotificationListenerService ─────────────────┐
│  监听通知 → 获取前台 App 包名 → 匹配场景配置     │
├─ AccessibilitySceneMode (AccessibilityService)─┤
│  通过无障碍服务捕获焦点变化 → 识别前台 App       │
├─ SceneTileService (QuickSettings Tile) ────────┤
│  用户手动在快捷设置面板切换（省电/均衡/游戏模式） │
├─ BroadcastReceiver (定时/条件触发) ─────────────┤
│  基于时间、电量、温度等条件自动切换              │
└────────────────────────────────────────────────┘
```

### 3.2 场景配置

场景配置存储在 `model/SceneConfigInfo.java` 中。每个场景定义了：

- 关联的应用列表
- 触发时的 CPU 调度策略
- GPU 频率/调度器
- 温度阈值
- 网络策略（如游戏时自动开启加速）

### 3.3 应用分类清单

`assets/categories.json` 中包含应用分类列表，涵盖了：

| 分类 | 示例应用 |
|------|----------|
| 短视频 | 抖音、快手、西瓜视频 |
| 视频应用 | 腾讯视频、爱奇艺、B站 |
| 游戏-高帧率 | 王者荣耀（120fps）、和平精英（90fps）|
| 游戏-原神 | 原神、崩坏（各渠道服）|
| 游戏-其他 | 绝区零、鸣潮、COD战区 |
| 地图导航 | 百度地图、高德地图、滴滴 |
| 购物 | 淘宝、京东、闲鱼、美团 |
| 系统应用 | MIUI 备份、小米换机 |

`refresh_rate.txt` 还分别对特定应用指定了期望帧率（如原神 60fps、王者荣耀 120fps）。

## 4. Xposed 模块

### 4.1 注册信息

- **入口**：`assets/xposed_init` → `com.omarea.xposed.XposedInterface`
- **Scope**：未限定具体包名（可能 Hook 系统框架）
- **minVersion**：未指定

### 4.2 模块组成

| 类 | 职责 |
|------|------|
| `XposedInterface` | Xposed 入口，`handleLoadPackage` 方法 |
| `XposedCheck` | 检测 Xposed 环境是否可用 |

### 4.3 可能的作用

Scene 作为 Xposed 模块的主要目标是**弥补 ROOT 权限的不足**：
- Hook 系统服务（如 ActivityManagerService）获取更精确的应用状态
- 修改系统属性（`SystemProperties.set()`）而不需要 ROOT
- 拦截系统调用实现更细粒度的控制

然而从代码量来看（仅 2 个文件），Xposed 功能相对精简，核心功能仍依赖 ROOT + kr-script。

## 5. 原生库（libnative-lib.so）

| 属性 | 值 |
|------|------|
| 架构 | ARM64 (aarch64) |
| NDK 版本 | r21（2020 年初）|
| 编译目标 | Android API 23+ |
| 符号状态 | **stripped**（无符号表）|
| 文件大小 | 较小（从目录未显示具体大小）|

### 交互方式

Java 端通过 `vtools/SceneJNI.java` 调用原生方法（`System.loadLibrary("native-lib")`）。从 JNI 类名推测，原生库提供了：

- **Binder 操作**：与 `toolkit/binder*.so` 配合，执行 Binder IPC 调用
- **系统调用**：绕过 Java 层限制直接执行 `ioctl`、`syscall` 等

> ⚠️ 由于 `.so` 文件已被 strip，无法直接获取符号名。如需深入分析需使用逆向工具（如 Ghidra/IDA Pro）反汇编。

## 6. 权限体系

Scene 申请的权限反映了其系统级工具的性质：

### 6.1 Android 权限

| 权限 | 用途 |
|------|------|
| `ROOT` | 最核心权限（非 Android 标准权限），通过 `su` 执行 Shell 脚本 |
| `WRITE_SECURE_SETTINGS` | 修改系统安全设置（adb grant 可获取）|
| `WRITE_SETTINGS` | 修改系统设置 |
| `SYSTEM_ALERT_WINDOW` | 悬浮窗显示 |
| `FOREGROUND_SERVICE` | 前台保活服务 |
| `RECEIVE_BOOT_COMPLETED` | 开机自启 |
| `REQUEST_INSTALL_PACKAGES` | 安装 APK |
| `PACKAGE_USAGE_STATS` | 获取应用使用统计 |
| `QUERY_ALL_PACKAGES` | 枚举所有安装的应用 |
| `BIND_ACCESSIBILITY_SERVICE` | 无障碍服务（场景模式）|
| `BIND_NOTIFICATION_LISTENER_SERVICE` | 通知监听（场景模式）|
| `WRITE_EXTERNAL_STORAGE` | 写入外部存储 |
| `MANAGE_EXTERNAL_STORAGE` | 管理所有文件 |

### 6.2 特殊机制

```
ROOT 权限 ─────→ kr-script 脚本（Shell 命令 sysfs 调校）
                   ↓
Xposed Hook ───→ 绕过 Java 层限制（修改系统属性等）
                   ↓
Accessibility ──→ 前台 App 识别 + 自动点击
                   ↓
Notification ───→ 前台 App 切换监听（备选方案）
```

## 7. 启动流程

```
系统启动完成
    │
    ▼
ReceiverBoot (BOOT_COMPLETED 广播)
    │
    ▼
BootService (开机服务)
    ├── 恢复已保存的配置（场景、脚本、Xposed 配置）
    ├── 启动保活服务 KeepAliveService
    └── 检查 Xposed 环境 → 初始化 XposedInterface
    │
    ▼
用户打开 App
    │
    ▼
ActivityStartSplash (启动屏)
    │
    ▼
ActivityMain (主界面)
    ├── 加载 kr-script 配置 → 构建功能菜单
    ├── 初始化监控服务（CPU/电池/FPS）
    └── 启动 NotificationListenerService
```

## 8. 混淆代码分析

### 8.1 范围

| 包 | 文件数 | 性质 |
|----|--------|------|
| `com.omarea.*` | 153 | **未混淆**（业务代码）|
| `a.*` | 3,334 | **ProGuard 混淆**（内部类被拍平）|
| `androidx.*` / `kotlinx.*` | 107 | 第三方依赖库 |

### 8.2 混淆影响评估

混淆对 `Scene.java` 和 `ActivityMain.java` 的影响示例：

```
Scene.java 导入的混淆类（约 48 个）：
  a.a42, a.a70, a.aa0, a.aq0, a.b62, a.b92, ...
  → 这些是原 Scene.java 的内部类或匿名类，
    被 ProGuard 拍平到 a/ 包

ActivityMain.java 导入的混淆类（约 30 个）：
  a.b62, a.bx1, a.d72, a.e50, a.e60, ...
  → 同理，原 ActivityMain 的内部类和匿名 Handler/Callback
```

**结论**：混淆代码主要来自**内部类和匿名类**被拍平到 `a/` 包。由于这些类的调用关系在非混淆代码中明确可追踪（`com.omarea.*` 引用 `a.*`），不影响理解**业务逻辑的主干流程**，但若要深入每个 Activity 的细节交互，则需要分析混淆代码。

### 8.3 恢复路线

如需还原混淆代码：

1. **追踪引用**：在非混淆代码中搜索 `import a.xxx`，确定每个混淆类的调用位置
2. **分析调用模式**：混淆类通常以 `a.xxx.method()` 形式被调用
3. **重命名映射**：基于调用上下文和参数类型推断原始名称
4. **工具辅助**：使用 ProGuard 映射文件（如果有 `mapping.txt`）可完全还原

## 9. 应用分类系统

`assets/categories.json` 是 Scene 用来识别应用所属类别的预置数据库：

```
格式：
{
  "friendly": "应用分类中文名",
  "packages": ["包名列表"],
  "category": "分类标识"
}
```

主要分类：

| category | 内容 | 数量（估计）|
|----------|------|-----------|
| ShortVideo | 短视频类 | ~10 |
| Video | 视频应用 | ~20+ |
| Game | 游戏（含子分类）| ~50+ |
| Map | 地图导航 | ~15+ |
| Shopping | 购物应用 | ~20+ |
| Social | 社交应用 | ~30+ |

这些分类用于场景模式中判断当前 App 应用什么性能策略。
